## Сильные стороны

1. **Архитектура с нулевым знанием** — сервер никогда не видит открытый текст
2. **Сильные криптопримитивы** — AES-256-GCM, Ed25519, X25519, BLAKE2b (всё через libsodium)
3. **Нет ошибок работы с памятью** — корректная проверка границ в C-коде, нет переполнений буферов
4. **Корректная очистка секретных ключей** — `sodium_memzero()` используется повсеместно
5. **Модель идентификации TOFU** — практичная и подходящая для данного сценария
6. **Разделение KDF** — разные контексты ("fearaudi"/"fearvide") и ID подключей для аудио/видео
7. **Эфемерные ключи ECDH** — новая пара ключей при каждом входе в комнату, `my_sk` обнуляется после использования
8. **Разрешения Android** — минимальные, без избыточных запросов
9. **Нет экспортированных компонентов** — только MAIN-активити экспортирована, сервисы правильно ограничены

## Уязвимости

### КРИТИЧЕСКАЯ

#### 1. Риск повторного использования nonce в AES-GCM для аудио/видео

**Расположение**: `AudioCallManager.kt`, `VideoCallManager.kt`, `audio_crypto.c`

Nonce формируется как `[prefix(4)][sequence(8)]`. Если звонок перезапускается с тем же ключом комнаты и случайный prefix совпадёт (парадокс дней рождения для 4 байт = ~65 тыс. звонков до 50% вероятности коллизии), одна и та же комбинация nonce+ключ зашифрует разные данные.

AES-GCM катастрофически ломается при повторном использовании nonce — атакующий может восстановить ключ аутентификации и подделывать пакеты.

**Смягчающие факторы**: Prefix генерируется заново при каждом звонке (4 случайных байта = 1 к 4 миллиардам на пару звонков), а ключи комнаты меняются между сессиями при использовании ECDH. Реальный риск для обычного использования низкий.

**Рекомендация**: Увеличить nonce prefix до 8 байт или использовать AEAD, устойчивый к повторному использованию nonce (например, AES-GCM-SIV).

### ВЫСОКАЯ

#### ~~2. Отсутствие защиты от MITM при обмене ключами ECDH~~ [ИСПРАВЛЕНО]

**Расположение**: `client.c` (ecdh_join_room / handle_key_request), `FearClient.kt`

~~Сообщение KEY_RESPONSE содержало `responder_pk` в открытом виде без подписи.~~

**Исправление**: Респондер подписывает свой эфемерный X25519 ключ долгосрочным Ed25519 ключом (`identity_sign` / `im.sign`). Входящий верифицирует подпись и выполняет TOFU-проверку. Реализовано на обеих платформах (десктоп + Android). Примечание: защита опциональна — требует наличия identity-ключей у обеих сторон.

#### 3. Отсутствие обнаружения replay-атак в UDP

**Расположение**: `audio_crypto.c`, `AudioCallManager.kt`, `VideoCallManager.kt`

Порядковые номера включаются в пакеты, но не проверяются на повторное воспроизведение:
```c
uint64_t seq = ntohll_u64(be_seq);
// Нет проверки, был ли seq уже виден
```

Атакующий может захватить и повторно воспроизвести аудио/видео пакеты, вызывая артефакты или путаницу.

**Рекомендация**: Реализовать скользящее окно обнаружения повторов (отслеживать последние 64 порядковых номера, отклонять дубликаты и пакеты старше окна).

### СРЕДНЯЯ

#### 4. Секретный ключ хранится открытым текстом

**Расположение**: `identity.c:111`, `IdentityManager.kt:228-237`

Секретный ключ Ed25519 записывается как открытый текст:
```kotlin
identityFile.writeText("PK:${base64Encode(pk)}\nSK:${base64Encode(sk)}\n")
```

- Десктоп: права файла 0600 на Linux, нет защиты на Windows
- Android: приватная директория приложения, но не зашифрована приложением

**Рекомендация**:
- Android: использовать `EncryptedSharedPreferences` или `AndroidKeyStore`
- Десктоп Linux: использовать системный keyring (Secret Service API)
- Десктоп Windows: использовать DPAPI

#### ~~5. Отладочное логирование конфиденциальной информации~~ [ИСПРАВЛЕНО]

**Расположение**: `FearClient.kt`

~~Ключи комнаты логируются на уровне INFO, видны через `adb logcat`.~~

**Исправление**: Все логи с ключами обёрнуты проверкой `BuildConfig.DEBUG` — в релизной сборке не выводятся.

#### ~~6. Захардкоженные реквизиты подписи в build.gradle.kts~~ [ИСПРАВЛЕНО]

**Расположение**: `app/build.gradle.kts`

~~Пароли были захардкожены в файле сборки.~~

**Исправление**: Пароли вынесены в `local.properties` и читаются через `localProps["STORE_PASSWORD"]` / `localProps["KEY_PASSWORD"]`.

#### 7. TOFU не обнаруживает split-brain атаки

**Расположение**: `identity.c:218-293`, `IdentityManager.kt:138-151`

Если атакующий регистрируется под тем же именем с другого места с другим ключом, система TOFU предупредит о смене ключа, но не сможет определить, какой ключ легитимный.

Отпечаток — всего 64 бита (8 байт хеша BLAKE2b) — достаточно для визуального сравнения, но коротко по современным стандартам.

**Рекомендация**: Задокументировать ограничения. Добавить QR-код или emoji-отпечаток для упрощения внеполосной верификации.

### НИЗКАЯ

#### ~~8. Содержимое файла не обнуляется перед освобождением памяти~~ [ИСПРАВЛЕНО]

**Расположение**: `client.c`

**Исправление**: `sodium_memzero(file_data, file_size)` вызывается перед `free()` во всех путях выполнения (ошибка чтения, ошибка шифрования, успешная отправка).

#### ~~9. Недостаточная валидация имён файлов~~ [ИСПРАВЛЕНО]

**Расположение**: `client.c`, `FearClient.kt`

~~Пути файлов из `/sendfile` не санитизировались.~~

**Исправление**: `basename()` применяется при отправке (извлечение имени файла) и при приёме (защита от path traversal) на обеих платформах. На Windows — ручной разбор `\\` и `/`.

#### 10. Нулевой nonce как маркер служебных сообщений

**Расположение**: `client.c:69-92`, `client.c:1020-1027`

Полностью нулевой nonce сигнализирует «не зашифровано». Вероятность коллизии пренебрежимо мала (1 к 2^96), но явный флаг был бы надёжнее.

#### 11. Отсутствие ограничения скорости на сервере

**Расположение**: `server.c`

Нет ограничения скорости сообщений по клиенту или комнате. Атакующий может заполнить комнату сообщениями, исчерпав пропускную способность других клиентов.

**Рекомендация**: Добавить ограничение сообщений на клиента (например, 100 сообщ/сек).

---

### ИНФОРМАЦИОННЫЕ (проектные компромиссы)

#### 12. Утечка метаданных

Сервер может наблюдать: членство в комнатах, время/частоту сообщений, приблизительные размеры сообщений, события передачи файлов. Открытый текст защищён, но метаданные — нет.

**Приемлемо для**: приватных серверов, доверенных сетей.
**Для высокой безопасности**: рассмотреть padding, фоновый трафик, луковую маршрутизацию.

#### 13. Отсутствие прямой секретности (Forward Secrecy)

Ключ комнаты статичен для всех сообщений в ней. При компрометации все прошлые и будущие сообщения в этой комнате могут быть расшифрованы.

**Приемлемо для**: временных/одноразовых комнат.
**Для высокой безопасности**: реализовать Double Ratchet (как в Signal Protocol) для эволюции ключей.

#### 14. Максимум клиентов сервера = 100

Захардкожено `MAX_CLIENTS=100` в `common.h`. Достаточно для малых групп, может потребоваться увеличение для масштабных развёртываний.

## Используемые криптографические примитивы

| Примитив | Стандарт | Применение |
|----------|----------|------------|
| AES-256-GCM | NIST SP 800-38D | Чат-сообщения, аудио, видео |
| Ed25519 | RFC 8032 | Подпись/верификация идентичности |
| X25519 | RFC 7748 | Обмен ключами ECDH |
| XSalsa20-Poly1305 | libsodium crypto_box | Транспорт ключей в ECDH |
| BLAKE2b | RFC 7693 | KDF для аудио/видео подключей |

Все криптографические операции используют **libsodium** — хорошо проверенную реализацию с постоянным временем выполнения.

## Заключение

F.E.A.R. Messenger демонстрирует качественную инженерию безопасности с хорошо спроектированной архитектурой нулевого знания и корректным использованием криптографических библиотек. C-код безопасен для памяти, переполнений буферов не обнаружено. Из 11 первоначально выявленных уязвимостей **5 исправлены** (#2, #5, #6, #8, #9), включая критически важную защиту ECDH от MITM-атак. Наиболее значительная оставшаяся проблема — отсутствие обнаружения повторных пакетов в UDP-потоках (#3). Остальные открытые проблемы (#1, #4, #7, #10, #11) относятся к категориям среднего и низкого приоритета. Система пригодна для безопасных коммуникаций в доверенных средах.
